# Repository Guidelines

## 项目总览

本项目是一个面向企业的“仿真顾客决策平台型 Agent”。产品首页应借鉴 AnyGen 的平台型 Agent 工作台范式：左侧承载工作区与资产导航，中间承载任务启动和能力入口，右侧承载仿真顾客心声预览。

产品核心不是通用聊天，而是让企业用一句话或一个模板发起“顾客决策仿真”，再由平台调度 Skill、Tool、资源库、知识库和仿真顾客完成决策预演。

## 核心产品主线

首页启动任务
→ 创建任务上下文
→ 选择或补充业务上下文
→ 拆解决策问题
→ 生成仿真顾客群体
→ 收集多视角反馈
→ 识别争议与风险
→ 生成方案优化建议
→ 输出决策报告
→ 沉淀项目与复盘记录

## 首页信息架构

首页分为三个核心区域。

左侧是平台资产与工作区：

* 顾客搭档
* 新建
* 探索技能
* 资源库
* 知识库
* 顾客群体库
* 项目
* 最近
* 我的

中间是任务启动区：

* 直接对话框：用于输入“今天要验证什么企业决策？”
* 能力按钮：辅助设计、统计分析、调研问卷、群体决策、爆款图谱。
* 场景按钮：定价决策、产品决策、营销增长、品牌定位、会员权益、服务政策。
* 模板：会员涨价评估、新品上市判断、广告文案顾客反应、品牌定位验证。

右侧是顾客心声区：

* 标题为“顾客正在想”。
* 只展示脱敏顾客信息和一句真实消费心理活动。
* 不放输入框、筛选器、分析标签或底部说明。

首页可以有对话框形态，但它必须是“任务启动器”，不是闲聊窗口。

## 首页到任务详情页的数据流

首页输入框不是静态跳转入口。用户在首页输入的文本必须创建一个任务上下文，并在任务详情页中作为第一段用户原始输入展示。

必须贯通的数据包括：

* 用户原始输入
* 选择的能力按钮，例如调研问卷、群体决策、统计分析
* 选择的业务场景，例如定价决策、产品决策、会员权益
* 引用的资源库条目
* 引用的知识库条目
* 本次任务调用的仿真顾客列表
* 后续生成的澄清问题、方案、顾客回复和结论

禁止在任务详情页写死与首页输入无关的第一段对话内容。任何任务详情页都必须能解释“这个任务从首页的哪次输入而来”。

## 任务详情页原则

任务详情页参考 AnyGen 任务页，但要适配“仿真顾客决策平台”的三栏结构：

* 左侧：收起态平台导航，只保留图标入口，不展开完整菜单。
* 中间：任务对话与结构化工作流，展示用户原始输入、Agent 澄清、方案生成、顾客回答进度和最终结论。
* 右侧：顾客留声区，与首页右侧风格一致，但内容切换为本次任务的顾客回答流。

任务详情页不是普通聊天页。中间区域必须随着任务推进呈现明确阶段，而不是只堆消息气泡。

### 当前任务详情页交互准则

当前前端原型中，默认任务类型为“调研问卷”。中间区域应采用“消息头像 + 结构化卡片”的形态：

* 用户原始输入使用用户头像和轻量消息气泡展示。
* Agent 澄清问题使用机器人头像和可折叠工作流卡片展示。
* 调研方案使用机器人头像和可折叠工作流卡片展示。
* 每个可折叠卡片右上角使用展开 / 收起图标，不额外增加复杂操作入口。
* 右侧顾客留声 hover 只保留一套弹窗机制，避免同时出现内置浮层和全局浮层。
* 顾客 hover 弹窗展示脱敏信息、顾客留言摘要、已购商品和仿真顾客人设，不展示支持 / 反对等分析标签。

## 深度访谈与圆桌模式

首页右侧“顾客正在想”的每一句仿真顾客心声都可以作为深度访谈入口。用户点击某位顾客的一句话后，应进入深度访谈页，并将该顾客在首页展示的原话作为访谈第一条消息。

深度访谈包含两个已确认形态：

1. 方案 A：一对一深访
   * 用于从首页顾客心声直接进入。
   * 页面重点是顾客消息、用户追问和右侧访谈对象档案。
   * 不展示“访谈目标”和“访谈关键问题”卡片。
   * 用户继续追问后，可提示是否进入“圆桌模式”。

2. 方案 C：顾客圆桌模式
   * 用户确认进入圆桌后，页面必须切换为独立圆桌工作台，而不是在方案 A 上简单增加右栏。
   * 顶部展示深度访谈任务名和来源顾客。
   * 第二行横向展示参与者：仿真顾客、用户 / 研究员、添加真人、添加仿真顾客、专家 Agent。
   * 主体左侧展示“访谈记录（圆桌对话）”。
   * 主体右侧展示“当前发言者档案”和“访谈结论草稿”。
   * 底部输入区用于继续追问、引用资源 / 知识、添加参与者。

深度访谈页不是普通私聊页。它的目标是把某个顾客心声展开为可追问、可多人共创、可沉淀结论的决策证据。

## 资源库与知识库定义

资源库是 Agent 的原材料，主要放结构化业务资产和素材资产，例如：

* 商品表、SKU 表、价格表
* 用户表、会员表、仿真顾客表
* 竞品表、渠道表、门店表
* 营销素材、设计素材、历史活动素材
* 问卷结果表、访谈记录整理表

知识库是 Agent 的背景理解，主要放企业知识、规则和长期上下文，例如：

* 品牌定位
* 用户研究报告
* 产品策略文档
* 会员体系说明
* 服务政策
* 价格策略原则
* 历史决策复盘
* 行业洞察
* 企业内部 SOP

## 顾客群体库定义

顾客群体库是可复用的仿真顾客 Panel，用于沉淀一组可反复调用的顾客。

示例：

* 会员涨价顾客群体
* 新品尝鲜顾客群体
* 价格敏感顾客群体
* 高价值会员顾客群体
* 线下门店顾客群体

单个仿真顾客画像从顾客群体库内部进入，不在一级导航放“我的 Persona”。“我的”只代表当前登录用户、企业空间、账号和设置。

## 探索技能页原则

探索技能页用于浏览、搜索、启用或停用当前工作台可用技能。

技能卡片应展示技能名称、说明、试用入口和使用次数。鼠标悬浮当前技能卡时，右上角使用“已启用 / 未启用”的开关样式，不使用“+ 添加”。

## 目标用户

主要用户包括企业创始人、产品负责人、市场负责人、品牌负责人、增长负责人、用户研究团队、咨询顾问和业务决策者。

用户不是技术开发者。前台产品应展示业务工作流和可调度能力，不暴露复杂技术编排。

## 产品不是什么

本产品不是：

* 通用 Chatbot
* 普通问卷工具
* 普通用户访谈记录工具
* 单纯数据看板
* 单纯市场调研报告生成器
* 只服务服装行业的爆款分析工具
* 只围绕商品基因的分析工具

## 产品是什么

本产品是：

* 企业决策前的 Agent 工作台
* 仿真顾客决策推演平台
* 多 Agent 决策编排系统
* 业务资源与知识调度平台
* 决策风险识别工具
* 决策建议与复盘系统

## 前台产品原则

首页应该像平台型 Agent 工作台，任务内页面应该像企业决策驾驶舱。

用户应该清楚看到：

* 当前可以调用哪些 Skill、Tool、资源和知识
* 当前要验证的企业决策是什么
* 系统如何拆解决策
* 仿真顾客是谁
* 不同顾客如何反馈
* 哪些人支持、反对、犹豫或条件接受
* 最大阻力、最大机会和关键风险是什么
* 决策应该继续、修改、测试还是暂缓
* 下一步应该做什么

实现 UI 时必须以已确认线框图为准。若当前代码与线框图在布局、信息层级、模块顺序或交互数据流上不一致，应优先修正实现，而不是用现有代码解释设计。

## Agent 架构原则

系统应采用垂直多 Agent 架构。推荐 Agent：

1. 决策主控 Agent
2. 决策拆解 Agent
3. 仿真顾客生成 Agent
4. 顾客反馈模拟 Agent
5. 争议与风险识别 Agent
6. 方案优化 Agent
7. 决策建议 Agent
8. 决策复盘 Agent

## 重要设计原则

1. 始终围绕“企业决策”组织产品。
2. 首页优先表达平台能力、资产和任务启动，不要一上来变成后台表单。
3. 对话框是任务入口，不是通用聊天体验。
4. 不要把产品做成只分析商品或爆款的工具。
5. 资源库和知识库必须区分清楚。
6. 仿真顾客必须表达认知、情绪、动机、顾虑、反对理由和行动倾向。
7. 每一个决策建议都要说明原因、支持证据和风险。
8. MVP 阶段可以使用 mock data，但数据结构必须接近真实业务。
9. 页面结构要清晰，适合企业用户快速理解。
10. 优先实现业务闭环，而不是复杂技术炫技。

## MVP 核心页面

第一版 MVP 包括八个核心页面：

1. 首页 / Agent 工作台
2. 探索技能详情页
3. 对话内容详情页 / 任务详情页
4. 深度访谈页 / 顾客圆桌模式
5. 决策输入页
6. 仿真顾客群体页
7. 决策推演与反馈页
8. 决策建议报告页

## 第一版验收标准

一个功能完成，需要满足：

* 用户能从首页发起企业决策仿真任务。
* 用户在首页输入的内容能原样进入任务详情页，并成为第一段用户原始输入。
* 用户能看懂当前可用的 Skill、Tool、资源库和知识库。
* 页面清楚展示仿真顾客的不同视角。
* 页面能区分支持、反对、犹豫和条件接受。
* 页面能展示决策风险。
* 页面能输出下一步行动建议。
* 产品看起来像平台型 Agent，而不是普通聊天机器人。
* 代码结构清晰，方便后续扩展。
